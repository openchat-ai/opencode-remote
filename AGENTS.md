## 代码规范

### JavaScript/Node.js
- **模块系统**: ESM (`import`/`export`)，禁止混用 CJS
- **缩进**: 2 空格
- **字符串**: 单引号优先. **分号**: 必须
- **命名**: camelCase 变量/函数，PascalCase 类
- **错误处理**: 禁止空 catch 块，必须记录日志
- **文件操作**: 统一使用 `fs/promises` 异步 API

### Git
- 提交前自动运行 husky pre-commit hook
- Commit message 格式: `type: description` (feat/fix/chore/refactor/docs)
- 禁止提交 `.env` 文件（已在 .gitignore 中）

## 专家点评系统

### 触发词: `叫全部专家` / `叫所有专家` / `呼叫专家点评` / `专家点评` / `专家意见` / `call all experts` / `expert review` / `z`

触发词前后可带用户自己的话。例如: `z kit怎么发版` 或 `帮我看看这个bug z`，专家会结合项目状态 + 用户问题一起回复。

### 流程（严格遵循）

```
1. 生成项目摘要（~300 字，不贴源码）
2. 提取用户问题（触发词前后的内容）
3. 启动 1 个 task，嵌入摘要 + 用户问题 + 14 角色
4. 1-13 号专家: 每人 2-3 句话点评，不投票
5. 技术经理: 根据 13 位意见，输出 1-15 共 15 条问题（不含投票），前 5 条 P0，中间 5 条 P1，最后 5 条 P2
6. 1-13 号专家: 每人从 15 条中投 3 票（格式：-> 三票：1, 7, 12）
7. 技术经理: 公布投票统计（得票：1(4) 3(6)...），汇入 P0-P2
```

用户看到的输出顺序：**13 条专家意见 -> 技术经理 15 条问题 -> 投票统计 -> 最终 P0-P2（含得票）**

### 自动执行 — 改进版流程

```
1. 生成项目摘要（~300 字，不贴源码）
2. 提取用户问题（触发词前后的内容）
3. 启动 1 个 task，嵌入摘要 + 用户问题 + 14 角色
4. 1-13 号专家: 每人 2-3 句话点评，不投票
   - **如果某条意见在上轮已提过但未修复，必须指出"上轮已提过，未落实"**
   - 格式: `[意见] 内容 / [上轮已提: N]`
5. 技术经理: 根据 13 位意见，输出 1-15 共 15 条问题
   - **每条必须标注对应的专家意见编号**（格式: `1) xxx [来源: 3,7,11]`）
   - 前 5 条 P0，中间 5 条 P1，最后 5 条 P2
   - **不含投票结果**
6. 1-13 号专家: 每人从 15 条中投 3 票
   - **投票时必须检查自己上轮的意见是否已落实，如未落实则优先投票给相关条目**
   - 格式：`-> 三票：1, 7, 12`
7. 技术经理: 公布投票统计，汇入 P0-P2，标记得票数
   - **格式**: `P0 🔴 1) xxx ( 票）[来源: 3,7] — 说明`
8. 执行后更新 @MEMORY.md:
   - 新增 `## 专家意见跟踪` 区块，每条意见记录:
     ```
     | 轮次 | 意见摘要 | 提出专家 | 对应任务 | 状态 | 验收人 |
     |------|---------|---------|---------|------|-------|
     | R1 | catch{}吞异常 | 安全研究员 | P0-5 | ✅ 已修(R2) | 安全研究员 |
     ```
   - 下轮专家评审前，先读取跟踪表，避免重复提相同意见
```

### 规则

- 1-13 号专家：**言辞必须苛刻、犀利、严谨。不讨好，不委婉。**
- **如果某条意见上轮提过且已标记完成，本轮不得再提**（除非验收不合格）
- 技术经理的 15 条问题必须标注来源专家编号，**没有对应来源的问题不得出现**
- **节约 token**：摘要不超过 400 字，不贴源码。prompt 直接用摘要嵌入，不额外传文件。专家输出限制 90 行。

### 四项基本原则（硬约束）

1. **一次性做好，不重复返工** — 同一个模块的同类问题最多修两轮。同类问题按功能特性标签归类（如"居民推理质量"），而非文件目录。第三轮如果还在提同类问题，说明方案不对，不是没修到位。此时技术经理必须输出的是"换方案"而不是"继续修"。触发"换方案"需≥2/3 专家投票确认旧方案不可行。第三轮技术经理的 15 条中必须包含"换方案"选项。

2. **以用户价值为导向** — P0 排序规则：
   - 第一优先级：用户能不能跑起来（onboarding 是否 5 分钟、CLI 是否可交互、核心功能是否可 demo）
   - 第二优先级：会不会崩（crash、数据丢失、安全漏洞）
   - 第三优先级：好不好用（性能、体验、文档）
   - 内部代码质量（目录结构、import 路径、测试风格）**默认不进 P0**，除非阻碍了前两项
   - 如果某轮 P0 全是内部质量项，没有用户可见项，技术经理必须指出并重新排序

3. **对用户友好** — 每次评审项目摘要必须包含一段"首次使用视角"：
   > 作为第一次使用的用户，当前最阻碍我的是什么：_________
   技术经理的 P0 必须至少有一条直接回应这个视角。
   **强制执行**：若 P0 列表没有条目直接回应首次使用视角，整轮评审无效，退回技术经理重出 15 条。每轮的首次使用视角文本必须有 git diff 可检测的实质性变更（不得复制上轮）。

4. **快速迭代，先上线** — 同一个项目连续评审超过 3 轮后（即 R4 开始），技术经理必须回答：
   > "当前是否可以发布一个可用版本？如果不行，最短路径是什么？"
   如果答案是"可以"，则本轮 P0 只保留阻塞发布的问题，其余降级或关闭。
   **目标是每 3 轮出一个可交付版本，而不是永远打磨。**
   **可交付版本验收标准**（全部满足才算"可交付"）：
   - CI 全绿（lint + test + build）
   - Docker 镜像可拉取
   - PM2 进程守护正常
   - 至少一个端到端用户场景可稳定跑通（P2P 语音或 AI 居民对话）
   - 新人 10 分钟内可完成一次完整体验
   - 无 P0 级安全漏洞
   **硬性版号**：R3 / R6 / R9 / R12 轮技术经理必须宣布版号（如 v0.1.0）。答不上来则本轮不得结束。

### 完成标准

一轮专家评审满足以下任一条件即可结束，不强制产出 P0：
- 跟踪表中全部意见标记已修，且本轮无新增 P0（票数 ≥ 3）
- 新增 P0 全是同一模块的重复问题 → 转"换方案"模式
- 连续两轮 P0 全部是内部质量项 → 跳过本轮，直接出发布版本

```
## 执行顺序

### 第 1 步：1-13 号专家点评，每人最多 200 字。**必须苛刻犀利。不说客套话。不投票。**

### 第 2 步：技术经理提出 15 条问题，基于以上 13 位意见，输出 1) 2) ... 15) 清单。
- **1-5 为 P0（红色🔴）**，6-10 为 P1（黄色🟡），11-15 为 P2（蓝色🔵）
- 每条包含一句话描述（做什么），不展开
- 小 bug 合并同类项，一条 = 一个可执行动作
- **不含投票结果。不含投票结果。不含投票结果。**
- 技术经理是操刀写代码的人，理解每个方向的代码实现路径。

### 第 3 步：1-13 号专家投票，每人从 15 条中选 3 条最关键的。格式：-> 三票：1, 7, 12

### 第 4 步：技术经理公布结果，得票统计，汇入 P0-P2，标记得票数。
- **P0 🔴 1) xxx ( 票）** - 说明
- **P1 🟡 7) xxx ( 票）**
```

### 摘要规范
- 只说事实，不说判断
- 每个模块标注状态：可用 / ⚠️ 半成品 / 死代码
- 列出已知 blocking bug
